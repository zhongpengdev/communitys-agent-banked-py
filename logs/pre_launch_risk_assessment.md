## 2. 风险点深度剖析

### 2.1 [P0] 缺乏自动化测试
*   **现状**：代码库中没有 `tests/` 文件夹，未发现任何单元测试或集成测试。
*   **事故场景**：在修复一个工具（Tool）的 Bug 时，无意中破坏了 JWT 校验逻辑。由于没有测试，该问题直到上线后用户无法登录才被发现。
*   **建议**：引入 `pytest`，针对 API 路由、JWT 校验和核心 Agent 逻辑编写基础测试用例。

### 2.2 [P0] 对话状态非持久化
*   **现状**：`app/services/agent_stream.py` 中使用 `InMemorySaver()` 作为 LangGraph 的 Checkpointer。
*   **事故场景**：服务器因更新或异常重启，所有用户的当前对话上下文（Agent 的记忆）全部丢失，用户发现 AI 变得“健忘”。
*   **建议**：改用持久化 Checkpointer（如 `langgraph-checkpoint-postgres`），将状态存入 Supabase 的 Postgres 数据库。

### 2.3 [P1] CORS 安全隐患
*   **现状**：`main.py` 中 `allow_origins=["*"]`。
*   **事故场景**：恶意网站可以通过脚本向你的后端发送请求，如果用户此时已在浏览器登录且后端未严格校验来源，可能导致越权操作。
*   **建议**：在生产环境中将 `allow_origins` 限制为前端应用的实际域名。

### 2.4 [P1] WebSocket 认证前置风险
*   **现状**：`app/api/dialog.py` 中先执行 `await websocket.accept()`，然后再校验 Token。
*   **事故场景**：攻击者可以建立数万个连接而不发送任何数据或发送垃圾数据，瞬间耗尽服务器的连接池和内存。
*   **建议**：在握手阶段（通过 Header 或第一个消息的快速校验）未通过时立即拒绝连接，或者限制单 IP 并发连接数。

### 2.5 [P1] 性能瓶颈：对象频繁初始化
*   **现状**：每次调用 `get_agent_response_stream` 都会重新创建 `ChatOpenAI` 和 `agent_executor`。
*   **影响**：增加了每次请求的初始延迟，且在并发较高时会造成显著的 CPU 和内存抖动。
*   **建议**：使用单例模式或依赖注入（DI）在应用启动时初始化这些对象。

### 2.6 [P2] 错误处理不统一
*   **现状**：部分接口手动返回 `{"code": 500, "message": "..."}`，且直接将 Python 的 `Exception` 字符串发送给前端。
*   **风险**：可能会在错误信息中泄露数据库字段名、代码逻辑路径等敏感信息。
*   **建议**：使用 FastAPI 的全局异常处理器（Exception Handler），统一返回格式，并区分“用户可见错误”和“内部日志错误”。

### 2.7 [P2] 数据类型隐患
*   **现状**：`user_id` 在 `authentication.py` 中是 `str`，但在 `session.py` 的部分接口中被声明为 `int`。
*   **风险**：在调用数据库或进行比较逻辑时可能触发 `TypeError`，导致请求失败。
*   **建议**：统一 `user_id` 的类型（建议使用 UUID 字符串或一致的长整型）。

## 3. 上线前建议操作清单 (Checklist)
1. [ ] **配置 RLS**: 确保 Supabase 数据库表开启了 Row Level Security，防止 Service Key 泄露导致的灾难。
2. [ ] **完善环境变量**: 将所有硬编码的 URL 和 API Key 迁移至 `.env`。
3. [ ] **引入结构化日志**: 将 `print` 替换为 `loguru` 记录，并设置日志文件滚动策略。
4. [ ] **限流机制**: 在 API 层（如 `slowapi`）或网关层增加限流，防止模型 API 额度被瞬间刷爆。
